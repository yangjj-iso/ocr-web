# 人社局需求功能测试记录

更新时间：2026-04-14  
测试口径：严格验收  
测试环境：本地开发环境，前端 `3000`，Java 控制面 `8080`，Python AI API `8001`，Worker metrics `9108`

## 测试样本

- 管理员账号：`admin`
- 样本分件任务：`rt_batch_20260414142718_afdf7c_eb836569`
- 样本批次：`batch_20260414142718_afdf7c`
- 样本页：`sf_batch_20260414142718_afdf7c_893358ba`

## 场景 1：基础可用性与认证

- 目标：验证服务是否能在登录后提供核心能力入口。
- 步骤：
  - 访问 `http://localhost:3000`
  - 登录 `POST /api/auth/login`
  - 调用 `GET /api/auth/me`
  - 调用 `GET /api/ocr/tasks?page=1&page_size=3`
  - 调用 `GET /api/archive/archive-records?page=1&page_size=3`
- 实际结果：
  - 前端首页返回 `200`
  - 登录成功
  - `auth/me` 返回 `200`
  - 任务列表返回 `200`
  - 档案列表返回 `200`
- 结论：`通过`
- 说明：基础环境可用，但“接口可用”不等于“满足需求”。

## 场景 2：单件图片 OCR 与字段沉淀

- 目标：验证系统是否已经把 OCR 结果与著录字段沉淀到档案记录。
- 步骤：
  - 抽查 `GET /api/ocr/tasks/search?q=报价&page=1&pageSize=3`
  - 抽查 `GET /api/archive/archive-records?page=1&page_size=3`
- 实际结果：
  - 任务搜索能返回带 `snippet` 的结果
  - 档案记录存在 `title/date/doc_no/responsible` 等字段，但质量不稳定，且部分字段为空
  - 样本 `storage_path` 为 `null`
- 结论：`部分通过`
- 说明：OCR 和字段写回存在，但质量与完整性达不到严格验收。

## 场景 3：页数自动统计与人工校对

- 目标：验证页数自动计算与人工修订能力。
- 步骤：
  - 读取分件任务详情
  - 读取 ReviewWorkbench 和 StructureReview 页面代码与现有 UI
- 实际结果：
  - 页面和任务模型存在 `page_count`
  - 分件页可显示页数区间
  - 未发现明确的“页数字段手工修正”入口
  - 未发现低置信度字符高亮
- 结论：`部分通过`
- 说明：自动统计存在，人工校对能力不足。

## 场景 4：归档章识别、表格识别与自动分件

- 目标：验证人社局最关键的“归档章识别 + 自动分件”链路。
- 步骤：
  - 调用 `GET /api/archive/tasks/rt_batch_20260414142718_afdf7c_eb836569`
  - 读取当前页 `ocr_pages`
  - 查看页面预览图与分件核对页实际展示
- 实际结果：
  - 样本页返回 `regions=[]`
  - 样本页只有大量 `lines`
  - 右侧面板展示为“识别行”，不是结构化区域
  - 表格未以 `table_data` 形式返回
  - 印章没有形成稳定的独立结构化结果
  - 当前预览图仍是倾斜扫描页
- 结论：`失败`
- 说明：当前样本无法证明自动分件链路达到验收要求，这一项按严格验收视为不通过。

## 场景 5：文件夹规则、路径回写与图片挂接

- 目标：验证 5 级目录、存放路径回写和图片自动挂接。
- 步骤：
  - 检查规则配置页面
  - 抽查 `archive-records` 的 `storage_path`
  - 搜索是否存在自动创建实体目录和自动搬运图片的稳定链路
- 实际结果：
  - 规则配置页面偏向“策略快照”，不是完整目录规则产品
  - 抽查 5 条档案记录，`storage_path` 全为 `null`
  - 未验证到自动创建 5 级实体文件夹
  - 未验证到按件号自动把图片挂接到实体目录
- 结论：`失败`
- 说明：这是流程性缺口，不是简单 UI 问题。

## 场景 6：全文检索与结果展示

- 目标：验证全文检索是否满足“档案层可查、结果可读、支持定位”。
- 步骤：
  - 调用 `GET /api/ocr/tasks/search?q=报价&page=1&pageSize=3`
  - 调用 `GET /api/ocr/tasks/search?q=重庆&page=1&pageSize=3`
  - 调用 `GET /api/archive/archive-records?page=1&page_size=3&q=报价`
  - 查看档案列表前端页面
- 实际结果：
  - 任务层搜索对 `报价` 和 `重庆` 有命中
  - 档案层搜索 `报价` 返回空结果
  - 档案列表页面没有关键词高亮
  - 档案列表没有稳定展示完整存放路径
  - 直接打开实体文件夹的能力未验证到
- 结论：`部分通过`
- 说明：当前更像“任务层搜索可用，档案层检索体验不达标”。

## 场景 7：角色权限与日志

- 目标：验证用户、角色和日志是否满足人社局安全要求。
- 步骤：
  - 检查 `AuthController`、`AuthService`、`router.js`、`authz.js`
  - 调用 `GET /api/admin/operation-logs?limit=5&offset=0`
- 实际结果：
  - 登录与角色存在：`sys_admin / tenant_admin / operator / searcher`
  - 后端接口有真实权限校验，不只是前端路由限制
  - 操作日志接口存在，但当前返回空列表
  - 现有日志更偏工作流动作，不能证明覆盖“查看、检索、著录修改、删除”等全部动作
- 结论：`权限通过，日志失败`
- 说明：权限链路成立，但完整审计留痕不成立。

## 场景 8：导出能力

- 目标：验证 Excel、CSV、双层 PDF 导出。
- 步骤：
  - 检查前后端导出接口
  - 调用 `POST /api/archive/batches/{batchId}/export/final-pdf`
  - 调用 `GET /api/ocr/batches/{batchId}/ai-merge-export`
- 实际结果：
  - Excel 导出能力存在
  - 未发现 CSV 导出接口
  - 批次 PDF 导出返回 `405`
  - AI merge 导出返回 `404`
  - 未验证到“原图层 + 隐藏文字层”的双层 PDF
- 结论：`失败`
- 说明：导出是当前严格验收中的硬缺口。

## 对人社局流程的测试判断

- 如果以“系统能否替代人工按流程完成著录、归类、检索、导出”为标准，当前结论是：`不能`
- 如果以“是否已经具备一个可继续开发的 OCR 工作流底座”为标准，当前结论是：`可以，但离验收级产品仍有明显距离`
