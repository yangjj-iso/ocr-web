## 流程验收总览

### 1. 著录前准备
- 纸质流程要求“整理、编号、加盖归档章并准确填写档号、年度、件号等信息”。
- 当前系统对“归档章检测和章内字段自动提取”仅具备局部能力，且当前分件页样本没有拿到结构化印章区域，不能作为稳定流程能力验收通过。

### 2. 扫描与文件组织
- 纸质流程要求“建立 5 级文件夹、扫描成图、按件号分件存储”。
- 当前系统支持批量上传、批次工作流和对象存储，但没有验收级别的“按规则自动建立实体文件夹并自动搬运图片”的能力。

### 3. 目录著录
- 纸质流程要求按文号、责任者、题名、日期、页数、密级等录入目录。
- 当前系统有字段提取、人工校对和档案记录，但缺 Excel 模板导入、批量修改和页数字段手工更正能力。

### 4. 全文检索
- 纸质流程要求 OCR、索引、关键字检索、文件夹定位。
- 当前系统具备任务层搜索与档案列表检索入口，但档案层关键字命中不稳定，且没有倒排索引和高亮展示。

### 5. 导出与安全
- 纸质流程要求导出著录数据、区分角色权限、记录日志、支持双层 PDF。
- 当前登录和角色基础存在，日志和导出能力都没有达到严格验收要求。

## 21 条需求逐项判定

### 01 模板导入
- 需求：支持导入 Excel 著录模板，字段至少包括文号、责任者、题名、日期、页数、密级、存放路径、备注。
- 代码证据：
  - 项目存在 Excel 导出与字段头定义，但没有对应的 Excel 模板导入接口或导入页面。
  - `frontend/src/api/archive.js` 未提供导入著录模板能力。
- 功能证据：
  - 已验证现有前端和接口没有可用的“导入著录模板”入口。
- 结论：`不满足`
- 风险：著录目录无法直接沿用人社局现有 Excel 模板流程。

### 02 自动识别填充
- 需求：OCR 自动提取文号、责任者、题名、日期、密级并填入著录表。
- 代码证据：
  - 存在 OCR、规则提取、LLM 提取和档案记录写回链路。
  - 分件/校对页面存在字段候选与元数据编辑能力。
- 功能证据：
  - 样本任务和档案记录中可见 OCR 文本、题名、日期等字段，但识别结果存在乱码、冗长题名和字段质量不稳定。
- 结论：`部分满足`
- 风险：字段提取存在，但结果质量和落表一致性不足，不能按严格验收判通过。

### 03 页数自动计算
- 需求：图片数量自动计为页数，并支持手工修正。
- 代码证据：
  - 当前任务、文档单元和档案记录都有 `page_count`。
  - 分件页能显示页数区间。
- 功能证据：
  - 样本数据里页数能够自动产生。
  - 未发现明确可用的“手工修正页数”入口。
- 结论：`部分满足`
- 风险：自动统计存在，但手工修正不完整。

### 04 批量修改
- 需求：支持按条件批量替换或补全责任者、密级等字段。
- 代码证据：
  - 当前归档数据编辑以单条记录/单个文档为主，未发现批量修改接口和 UI。
- 功能证据：
  - 前端没有批量著录修改操作。
- 结论：`不满足`
- 风险：大批量历史目录治理需要人工逐条处理。

### 05 人工校对界面
- 需求：高亮低置信度字符，支持逐件或逐字段修改，并记录修改日志。
- 代码证据：
  - 存在 ReviewWorkbench、StructureReview、CatalogReview 等人工页面。
  - 存在字段表单和候选值采纳能力。
  - 未看到低置信度字符高亮链路，也未看到逐字段修改日志链路。
- 功能证据：
  - 当前人工校对页面可以改字段，但没有低置信度高亮。
  - 当前日志接口没有体现逐字段修改明细。
- 结论：`部分满足`
- 风险：有人机校对外壳，但缺严格验收要求中的“置信度提示 + 变更可追溯”。

### 06 著录表导出
- 需求：支持 Excel、CSV，或通过 API 推送至数字档案管理系统。
- 代码证据：
  - 存在 Excel 导出链路。
  - 未发现 CSV 导出接口。
  - 未发现向外部数字档案管理系统推送的稳定接口。
- 功能证据：
  - 当前可见 Excel 能力；CSV 和 API 推送未提供可测入口。
- 结论：`部分满足`
- 风险：仅覆盖部分导出场景。

### 07 规则配置
- 需求：管理员可配置档号组成规则与分隔符。
- 代码证据：
  - `RulesConfigPage.vue` 只提供策略快照与 JSON 规则编辑。
  - 没有和 5 级目录规则、文件夹命名、实体路径生成形成一体化产品配置。
- 功能证据：
  - 前端可查看和编辑策略快照，但不等同于满足流程文件中的目录建档规则配置。
- 结论：`部分满足`
- 风险：有规则管理雏形，但不是可落地的归类规则引擎管理台。

### 08 自动建文件夹
- 需求：根据规则批量生成 5 级文件夹。
- 代码证据：
  - 未发现稳定的物理目录创建服务、目录任务和目录回填闭环。
- 功能证据：
  - 当前样本档案记录 `storage_path` 全为 `null`。
- 结论：`不满足`
- 风险：与纸质流程“先建目录再存放文件”直接不符。

### 09 图片自动挂接
- 需求：根据件号自动将图片移动或复制到对应件号文件夹。
- 代码证据：
  - 批量上传存在。
  - 未发现按件号自动挂接到实体目录的稳定实现。
- 功能证据：
  - 当前系统以批次/对象存储为主，不是按验收要求的实体件号目录管理。
- 结论：`部分满足`
- 风险：上传能力存在，但挂接与归位能力不足。

### 10 归档章识别与自动分件
- 需求：识别归档章、提取章内字段、自动分件、无法识别转人工队列。
- 代码证据：
  - `app/core/ocr_engine.py` 具备 `seal` 识别、表格提取、结构化区域输出能力。
  - 但当前分件页样本在接口返回中只有 `lines`，没有 `regions/table_data`。
- 功能证据：
  - 样本分件任务 `rt_batch_20260414142718_afdf7c_eb836569` 的页面数据 `regions=[]`。
  - 右侧面板展示的是识别行，不是结构化区域；表格没有被识别成表格，印章也没有以稳定的结构化结果呈现。
  - 当前预览图仍是倾斜扫描页，不是强文档矫正结果。
- 结论：`高风险待确认`
- 严格验收解释：按当前样本表现，视同`不通过`。
- 风险：这是自动分件主链路的 P0 问题。

## 对人社局流程的最终判断

- 当前系统更像“面向 OCR 处理和档案工作流的半成品平台”，不是“可以按人社局现行著录与归档流程直接替代人工”的验收级产品。
- 如果现在直接拿去做人社局严格验收，结论会是：
  - 著录辅助：有能力但不完整
  - 自动归类：未形成稳定流程
  - 全文检索：基础可用但不达标
  - 安全审计：架子存在但证据不足
  - 导出归档：关键输出物缺失
